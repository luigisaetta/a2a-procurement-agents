# Procurement Persistent Data Model Specification

Version: 0.1.0

Status: Draft

---

# Purpose

This specification defines the persistent data entities used by the A2A Procurement Agents platform.

Agents remain independently deployable black boxes and continue to communicate through A2A contracts. Persistent data provides durable business state, auditability, traceability, and resumable workflow foundations.

This document defines the logical data model. Physical database DDL, indexes, partitions, retention policies, and Oracle-specific implementation details will be specified in later iterations.

---

# Design Principles

The persistent data model must:

- preserve agent independence
- avoid exposing one agent's internal runtime state to another agent
- store business facts and workflow evidence, not framework internals
- support auditability and explainability
- support future checkpointing and resumable workflows
- make external system references explicit
- keep schema names stable and business-oriented

Agents may persist and read shared business entities, but they must not depend on another agent's private implementation tables or in-memory structures.

---

# Entity Groups

The initial data model is organized into these groups:

- procurement demand
- supplier master and sourcing data
- bid collection data
- offer evaluation data
- compliance data
- purchase order data
- workflow and audit data

---

# Common Attributes

All persistent entities should include these attributes unless explicitly stated otherwise:

| Attribute | Type | Required | Description |
| --- | --- | --- | --- |
| `created_at` | timestamp | yes | Timestamp when the record was created. |
| `created_by` | string | yes | Agent, system, or user that created the record. |
| `updated_at` | timestamp | yes | Timestamp when the record was last updated. |
| `updated_by` | string | yes | Agent, system, or user that last updated the record. |
| `version` | integer | yes | Optimistic concurrency version. |

Identifiers should be stable strings generated by the owning process or external system. UUIDs may be used internally, but externally meaningful IDs such as `request_id`, `supplier_id`, `offer_id`, and `purchase_order_id` remain first-class attributes.

# Procurement Demand

## ProcurementRequest

Represents the top-level procurement demand to be fulfilled.

| Attribute | Type | Required | Description |
| --- | --- | --- | --- |
| `request_id` | string | yes | Unique procurement request identifier. |
| `request_status` | enum | yes | Current status: `draft`, `sourcing`, `evaluating`, `approved`, `ordered`, `cancelled`, `failed`. |
| `request_priority` | enum | yes | Business priority: `standard`, `urgent`, `critical`. |
| `plant_code` | string | yes | Requesting plant, site, or business unit. |
| `requested_by` | string | no | User or system that initiated the request. |
| `currency` | string | yes | Expected ISO 4217 currency for offer comparison. |
| `evaluation_policy_id` | string | yes | Policy identifier to use for offer evaluation. |
| `response_deadline` | timestamp | no | Deadline for supplier bid responses. |
| `business_justification` | string | no | Optional reason for the procurement request. |
| `external_reference` | string | no | Reference to an upstream ERP, ticket, work order, or maintenance request. |

Primary key:

- `request_id`

Written by:

- Procurement Orchestrator

Read by:

- Bid Collection Agent
- Offer Evaluation Agent
- Compliance Agent
- Purchase Order Agent

# Initial Relationships

The initial relationship chain is:

1. `ProcurementRequest` has many `ProcurementRequestPart` records.
2. `Supplier` has many `SupplierCapability` and `SupplierEndpoint` records.
3. `ProcurementRequestPart` has many `SupplierSelection` records.
4. `SupplierSelection` may produce one `SupplierBidRequest`.
5. `SupplierBidRequest` may produce one `SupplierBidResponse`.
6. `SupplierBidResponse` may produce one `SupplierOffer`.
7. `SupplierOffer` records are evaluated by an `OfferEvaluationRun`.
8. `OfferEvaluationRun` produces one `OfferEvaluationDecision`.
9. A selected `SupplierOffer` may produce one `PurchaseOrderLine`.
10. `PurchaseOrderLine` belongs to one `PurchaseOrder`.
11. `ProcurementWorkflowRun`, `AgentTaskRecord`, and `AuditEvent` provide cross-agent traceability.

# Deferred Entities

These entities are intentionally deferred until the workflow needs them:

- contract and pricing agreements
- supplier negotiation messages
- user approval tasks
- shipment and delivery tracking
- invoice and payment records
- inventory reservations
- policy version history
- authentication secret storage
